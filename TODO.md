# 待办事项清单

## 架构说明

### 1. 架构目标

- 构建一个可公网访问的网页端 MVP
- 支持上传长聊天截图或粘贴聊天文本
- 由后端统一执行 OCR、规则映射、AI 分析与结果存储
- 前端只负责输入、展示和历史查看

### 2. 技术选型

- `Frontend`: `Vue 3`
- `Backend`: `Go`
- `Database`: `PostgreSQL`
- `AI Provider`: `Kimi API`
- `Deployment`: `Docker`
- `Persistent Storage`: `Docker Volume`

### 3. 系统模块

#### 3.1 Frontend

- 提供输入页面
- 提供结果展示页面
- 提供历史记录页面
- 通过 HTTP/JSON 调用后端 API

#### 3.2 Backend API

- 接收图片上传和文本输入
- 调用后端内部 OCR 能力
- 调用分析编排层
- 读写 PostgreSQL
- 管理上传文件

#### 3.3 OCR 能力层

- 仅存在于后端内部
- 负责对聊天截图进行识别
- 输出结构化消息数据
- 不允许前端直接调用 OCR 服务

#### 3.4 分析编排层

- 负责把 `chat-skills` 规则转成程序化执行步骤
- 负责拼装发给 `Kimi API` 的提示词
- 负责约束模型输出为稳定的结构化结果
- 负责分析流程编排：
  - 图片输入时：`图片 -> 预处理 -> OCR -> 结构化 -> 分析`
  - 文本输入时：`文本 -> 清洗 -> 结构化 -> 分析`

#### 3.5 Analysis Engine

- 基于 `chat-skills` 规则做阶段识别
- 做参数量化
- 计算 `IVI`、`SPE`、`EWS`
- 生成策略结论
- 将规则计算结果与 `Kimi API` 生成结果合并为最终报告

#### 3.6 Data Storage

- `PostgreSQL` 保存分析记录、结构化消息、结果摘要、时间戳
- `Docker Volume` 持久化上传图片和中间文件

### 4. 请求处理链路

#### 4.1 图片分析链路

1. 前端上传长聊天截图到后端
2. 后端执行图片预处理
3. 后端 OCR 能力层提取聊天内容
4. 后端将 OCR 结果转成结构化消息
5. 分析编排层加载 `chat-skills` 规则
6. 分析引擎完成阶段识别、参数量化、指标计算
7. 编排层组装 Prompt 并调用 `Kimi API`
8. 后端校验并存储结果
9. 前端展示分析报告

#### 4.2 文本分析链路

1. 前端提交聊天文本到后端
2. 后端执行文本清洗和结构化
3. 分析编排层加载 `chat-skills` 规则
4. 分析引擎完成阶段识别、参数量化、指标计算
5. 编排层组装 Prompt 并调用 `Kimi API`
6. 后端校验并存储结果
7. 前端展示分析报告

### 5. 容器部署模型

- `nginx` 容器：负责 HTTPS、静态资源分发、反向代理
- `frontend` 构建产物：由 `nginx` 提供访问
- `backend` 容器：运行 Go API、OCR 调度、分析编排逻辑
- `postgres` 容器：保存业务数据
- `docker volume`：保存上传图片和中间文件

### 6. 关键设计约束

- 不允许把 `chat-skills` 规则直接原样交给 `Kimi API` 自由发挥
- 必须由后端先做规则编排、特征提取和指标计算
- OCR 逻辑必须放在后端内部能力层
- 上传文件必须持久化到 `Docker Volume`
- 分析流程必须有单独的后端编排层，不能把所有逻辑塞进路由处理函数


## 开发待办

1. 拆解 `chat-skills` 规则体系  
把 `SKILL.md`、`input-rules.md`、`data-quantization-sop.md`、`algorithm-weight.md`、`output-rules.md`、`stage-model.md` 整理成程序可执行逻辑。

2. 设计分析流水线  
形成固定链路：`输入内容 -> 预处理 -> OCR/文本结构化 -> 特征提取 -> 参数量化 -> 核心指标计算 -> 策略生成 -> 结果生成`

3. 设计阶段识别逻辑  
基于 `stage-model` 输出当前互动阶段，如初识、升温、舒适区、邀约窗口、冷淡/冲突等。

4. 设计核心指标体系  
首版至少实现并展示：
- `IVI` 意图真实度
- `SPE` 社交势能
- `EWS` 升温窗口

5. 设计结果输出结构  
结果页统一输出：
- 对方心理/态度倾向
- 当前互动阶段
- 核心指标结论
- 下一步聊天建议
- 可直接发送的话术参考

6. 设计参数面板方案  
作为唯一保留的可选项，评估是否展示 `Sp`、`Fback`、`Pface`、`Rlatency` 等细粒度参数，以及展示方式。

7. 实现输入模块  
支持两种输入：
- 上传单张长聊天截图
- 粘贴聊天文本

8. 实现图片预处理  
对长截图做基础预处理，保证后端 OCR 与分析链路可用。

9. 在后端实现 OCR 与结构化解析能力层  
把截图提取为消息内容，并尽可能识别文本、消息顺序、时间特征、语气特征。OCR 作为后端内部能力，由分析编排层统一调度，不作为前端直连服务。

10. 实现文本直分析  
纯文本输入可直接进入分析，不依赖图片识别。

11. 实现分析编排层与分析引擎  
由后端统一完成规则映射、OCR 调度、特征提取、阶段判断、参数量化、核心指标计算、提示词组装、Kimi 调用、结构化结果校验和策略生成，避免把 `chat-skills` 规则直接裸传给模型自由发挥。

12. 实现聊天建议生成  
给出 2～3 条建议，覆盖推进、观察、收缩等不同策略。

13. 实现回复文案生成  
输出几条自然、可直接参考的回复话术。

14. 实现基础历史记录  
保留最近分析结果，方便回看。

15. 实现结果展示页  
以网页应用形式展示完整分析结果，适合桌面端浏览和远程演示。

16. 加入免责声明与基础风控文案  
在页面中加入“仅供沟通参考，不构成心理诊断”等说明。

17. 完成云端部署方案  
准备生产环境配置、部署方式、公网访问方案，明确前端静态资源、Go 后端、PostgreSQL、文件存储卷的容器化关系。

18. 部署 Demo 到云服务器  
完成上线并确保可通过网址远程访问，使用 Docker 镜像部署，上传文件通过 Docker Volume 持久化。
